## Overview

This document is the canonical artifact reference for the `egohygiene/reflector` publication
system. It documents every publication artifact: its producer, consumer, location, lifecycle
stage, publication destinations, and archival targets.

For the broader publication architecture, see
[`docs/publication-system-reference.md`](publication-system-reference.md).

---

## Artifact Lifecycle Stages

Each artifact passes through one or more of these lifecycle stages:

| Stage | Description |
|---|---|
| **Source** | Static file committed to the repository |
| **Build** | Generated by a build process (LaTeX, scripts) |
| **Staged** | Assembled into the release staging directory |
| **Deployed** | Published to GitHub Pages |
| **Released** | Uploaded as a GitHub Release asset |
| **Archived** | Deposited in a long-term archive (Zenodo, arXiv) |

|---|---|---|

### Canonical arXiv Bundle

Contents of `reflector-arxiv-vX.Y.Z.zip` / `.tar.gz`:

```
arxiv/
├── paper.tex               (root LaTeX entry point)
├── references.bib          (bibliography)
├── .latexmkrc              (build configuration)
├── 00README.json           (arXiv submission manifest)
├── sections/               (semantic content files)
├── figures/                (all figure assets)
├── diagrams/               (diagram files)
├── assets/                 (additional assets)
├── references/             (reference files)
├── styles/                 (style sheets)
├── macros/                 (macro definitions)
└── config/                 (configuration files)
```

The arXiv bundle is deterministic and reproducible:
- `tar` uses `--sort=name` and epoch `--mtime`
- `zip` uses `-X` to strip extended attributes

Validated by `scripts/validate-arxiv-packaging.py` (31 checks).

## Checksums and Manifest Contract

`scripts/stage-publication-release.py` is the canonical release inventory generator.

It:
1. Validates the required staged artifacts by expected path
2. Generates `checksums.txt` with SHA-256 hashes in deterministic filename order
3. Generates `release-manifest.json` with artifact inventory, checksum inventory, and release URLs
4. Generates `publication-inventory.json` with canonical artifact listing and ownership metadata
5. Embeds DOI metadata (version DOI + concept DOI) for publication/citation tooling
6. Fails with explicit expected-path diagnostics when a staged artifact is missing

**Smoke test:** `pytest -q tests/test_stage_publication_release.py`

## DOI Synchronization

The canonical DOI fields are sourced from `metadata/publication.yaml` and must be synchronized
across all publication metadata surfaces:

| DOI Field | Canonical Value |
|---|---|
| Version DOI | `10.5281/zenodo.20477044` |
| Concept DOI | `10.5281/zenodo.20477045` |

**Surfaces requiring DOI synchronization:**

| Surface | Field(s) |
|---|---|
| `metadata/publication.yaml` | `identifiers.doi`, `doi_url`, `zenodo_concept_doi`, `zenodo_concept_doi_url` |
| `CITATION.cff` | `doi`, `url`, `preferred-citation.doi` |
| `.zenodo.json` | `doi`, `conceptdoi` |
| `codemeta.json` | `identifier` |
| `publication.json` | `future.doi_generation.*` |
| `release-manifest.json` | `future_integrations.doi.*` |

Enforced by `scripts/validate-metadata.py`. DOI drift is a release-blocking validation failure.
